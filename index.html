Middleware and Service

Adaptability Lab: Cloud and Autonomic Computing

The general purpose of this lab was to enhance a knowledge of concepts and technologies for virtualization techniques. These techniques are used in IT environments that support the provisioning (i.e. development, deployment, management) of novel software systems (with their potential constraints such as QoS and security). These environments are typically dynamic and distributed.

The objectives were:

  • Discover virtualization archetectures and technologies
  • Recognize the fundamental differences between the two main types of virtualisation hosts, i.e. virtual machines (VM) and containers (CT).
  • Operate VirtualBox VMs provisioning, in NAT mode, and make appropriate configurations so that VMs can access and be accessed through the Internet.
  • Operate Docker containers provisioning, and make appropriate configurations so that containers can access and be accessed through the Internet.
  • Operate proper VMs provisioning with OpenStack and manage the networking connectivity.
  • Implement a simple orchestration tool in order to provision a Web 2-tier application using node.js services

Middleware for IoT Lab

During this lab we discovered three mainly used tools in IoT. Firstly, we used MQTT protocol to compute a LED. To do that, we set up a moquitto broker and publishing subscribing topic with an arduino.

Then, we discovered the restfull OM2M standard. It's a platform that enables us to set up a ressource archetecture that contains sensors but also their data/states. Thus, we can interact with this ressources via the API Rest queries.

Finally, linked these two tools thanks to Node-Red. Node-RED is a programming tool for wiring together hardware devices, APIs and online services in new and interesting ways.

Service Oriented Architecture: Room management

We realized in a group of 3 students a project of Room’s Management to apply Service Oriented Architecture’s and Middleware lessons. The goal was to realize a Web application to manage a room, like opening the door, switching on/off the light depending on the state of a samples of sensors. We had to implement diffrent scenarios based on the state of the sensor. As example of scenario: "close the windows when the temperature is under 18°C"

Jira: How we organized our project

To do that, we worked in agile using the tool Jira. It is a project management tool allowing us to work in agile. Consequently, we organized the project in 3 sprints composed of userstories correponding to the different scenarios.

OM2M Platform

In order to monitor the sensors we set up an OM2M Platform which have all the ressources needed. This platform has the following tree archetecture of ressources:

We organized the elements by platforms AE (application entity). In each platform there are the sensors and actuators containers. Each sensor/actuator contains its data CIN.

This platforms were created by a java program by using POST requests, it was easier than using a rest client (i.e Postman). In fact, if the OM2M platform is destroyed, it is easier to rebuild it just by launching this program. It is really faster than resend all request in a REST client.
